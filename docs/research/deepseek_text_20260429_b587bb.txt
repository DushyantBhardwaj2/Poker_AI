## 5. Development Roadmap (Phase-wise)

### Phase 1: Planning & Setup
- [ ] Finalize requirements and tech stack decisions
- [ ] Create GitHub repository (main, develop branches)
- [ ] Set up frontend folder (Next.js + Tailwind)
- [ ] Set up backend folder (Node.js + Express + MongoDB)
- [ ] Set up ML folder (Python + FastAPI scaffold)
- [ ] Configure environment variables template

### Phase 2: Core Backend Development
- [ ] Design database schema (Users, Games, Players, Actions, HandHistory)
- [ ] Implement game engine:
  - [ ] Create game with blinds/antes
  - [ ] Add/remove players with stack sizes
  - [ ] Deal hole cards (random, user-set, or manual)
  - [ ] Manage community cards per round
- [ ] Build action tracker:
  - [ ] Validate legal actions (fold, check, call, raise, all-in)
  - [ ] Automatically calculate pot (main & side pots)
  - [ ] Advance dealer button & turn order
- [ ] Implement REST API endpoints:
  - [ ] `POST /api/game/create`
  - [ ] `POST /api/player/add`
  - [ ] `POST /api/cards/set`
  - [ ] `POST /api/action/add`
- [ ] Add JWT authentication (optional for MVP)

### Phase 3: Frontend Development
- [ ] Build setup screen: game configuration, player seats, card selection
- [ ] Build live tracker UI:
  - [ ] Visual table layout
  - [ ] Player info (stack, cards if shown)
  - [ ] Community cards display
  - [ ] Pot display
- [ ] Action selector component (fold, check, call, raise slider, all-in)
- [ ] Real-time state updates via API polling (or WebSocket later)
- [ ] Basic dashboard for hand history (list of past hands)

### Phase 4: AI/ML Integration
- [ ] Create Monte Carlo simulation service (FastAPI):
  - [ ] Input: known hole cards, community cards, number of opponents, dead cards
  - [ ] Output: win/tie probability with confidence interval
  - [ ] Early stopping when 95% CI width < 1%
- [ ] Collect/generate training data (synthetic or from public hand histories)
- [ ] Train bluff detection model (using bet sizing, timing, board texture)
- [ ] Build opponent classification (clustering based on VPIP/PFR/AF)
- [ ] Develop recommendation engine:
  - [ ] Compute pot odds
  - [ ] Blend ML predictions with mathematical odds
  - [ ] Output Fold/Call/Raise/All-in with confidence
- [ ] Expose AI endpoints:
  - [ ] `POST /api/ai/suggest` → move & bet size
  - [ ] `POST /api/ai/bluff` → bluff probability
  - [ ] `POST /api/ai/opponent` → player type

### Phase 5: Testing
- [ ] Unit tests for game engine (pot calculation, side pots, all-in logic)
- [ ] Integration tests for API flows
- [ ] Unit tests for Monte Carlo simulation (statistical correctness)
- [ ] Edge cases: split pots, disconnected players, invalid actions
- [ ] Load testing on AI endpoints (ensure < 200ms response)

### Phase 6: Deployment
- [ ] Deploy frontend to Vercel (with env vars)
- [ ] Deploy backend to Railway/Render (auto-deploy on merge to main)
- [ ] Deploy ML service (separate FastAPI app on Render or HuggingFace)
- [ ] Set up CI/CD with GitHub Actions (lint, test, build, deploy)
- [ ] Monitoring & logging (e.g., Sentry, Logtail)

### Phase 7: Future Scope
- [ ] Screenshot OCR integration for automated input
- [ ] Voice commands (Web Speech API)
- [ ] Real-time overlay via Electron app
- [ ] Multiplayer sync (websocket server, cloud game state)
- [ ] GTO solver integration for Nash equilibrium advice
- [ ] Monetization: premium strategies, coaching sessions
